# LLM 위키 — LLM으로 키우는 개인 지식베이스 패턴

> 🧩 Andrej Karpathy가 공유한 "LLM Wiki" 노트는 RAG처럼 매 질문마다 원문을 다시 뒤지는 대신, **LLM이 상호링크된 마크다운 위키를 점진적으로 짓고 유지하게 하는** 패턴을 제안합니다. 원문(raw) · LLM이 소유하는 위키 · 스키마(CLAUDE.md)의 3계층 위에서 Ingest·Query·Lint 세 가지 운영을 반복하면, 지식이 매번 재발견되지 않고 **하나의 복리(compounding) 자산으로 축적**됩니다. 사람은 큐레이션과 좋은 질문을, LLM은 요약·상호참조·정리 같은 잡일을 맡습니다.

## 1. 문제: RAG는 지식을 축적하지 않는다

대부분의 사람이 LLM으로 문서를 다루는 방식은 **RAG**입니다. 파일 뭉치를 올려두면 질문할 때마다 관련 조각을 검색해 답을 만듭니다. 잘 동작하지만 한 가지 근본 한계가 있습니다.

> ⚠️ RAG는 매 질문마다 **지식을 처음부터 다시 발견**합니다. 문서 다섯 개를 종합해야 답할 수 있는 미묘한 질문이라면, LLM은 매번 그 조각들을 찾아 이어 붙여야 합니다. **아무것도 쌓이지 않습니다.**

NotebookLM, ChatGPT 파일 업로드, 대부분의 RAG 시스템이 이 방식입니다. 매번 바닥부터 시작하므로 탐구가 반복될수록 낭비가 커집니다.

## 2. 핵심 아이디어: 위키를 복리 자산으로

이 노트가 제안하는 방식은 다릅니다. 질문 시점에 원문을 검색하는 대신, **LLM이 지속적인 위키(persistent wiki)를 점진적으로 짓고 유지**합니다. 위키는 당신과 원문 사이에 놓인, 구조화되고 서로 링크된 마크다운 파일 모음입니다.

새 소스를 추가하면 LLM은 단순히 색인만 하지 않습니다.

- 소스를 **읽고** 핵심 정보를 추출한다.
- 기존 위키에 **통합**한다 — 엔티티 페이지 갱신, 주제 요약 수정.
- 새 데이터가 기존 주장과 **모순되면 표시**한다.
- 진화하는 종합(synthesis)을 강화하거나 반박한다.

즉 지식은 **한 번 컴파일되고 그 뒤로는 계속 최신 상태로 유지**됩니다. 매 질문마다 다시 유도하지 않습니다.

> 🌱 핵심 차이: 위키는 **지속적이고 복리로 불어나는 산출물**이다. 상호참조는 이미 걸려 있고, 모순은 이미 표시돼 있으며, 종합은 지금까지 읽은 모든 것을 이미 반영한다. 소스를 더할 때마다, 질문을 던질 때마다 위키는 더 풍부해진다.

## 3. 역할 분담: 사람은 큐레이션, LLM은 유지보수

당신은 위키를 (거의) 직접 쓰지 않습니다. **LLM이 위키 전체를 쓰고 유지**합니다.

| 주체 | 역할 |
|------|------|
| **사람** | 소싱(무엇을 넣을지 큐레이션), 탐색, 방향 제시, **좋은 질문 던지기** |
| **LLM** | 요약, 상호참조, 파일링, 기록(bookkeeping) — 지식베이스를 쓸모 있게 유지하는 모든 잡일 |

Karpathy의 실제 작업 모습이 이 분담을 잘 보여줍니다.

> 🖥️ 한쪽에는 LLM 에이전트, 다른 쪽에는 Obsidian을 띄운다. 대화에 맞춰 LLM이 위키를 편집하면, 나는 실시간으로 결과를 훑는다 — 링크를 따라가고, 그래프 뷰를 확인하고, 갱신된 페이지를 읽으며. **Obsidian은 IDE, LLM은 프로그래머, 위키는 코드베이스다.**

## 4. 아키텍처: 3계층

이 패턴은 세 개의 계층으로 구성됩니다.

1. **Raw 소스 (원문)** — 큐레이션한 원본 문서 모음. 기사·논문·이미지·데이터 파일. **불변(immutable)** 이다. LLM은 읽기만 하고 절대 수정하지 않는다. 여기가 **진실의 원천(source of truth)**.
2. **위키 (Wiki)** — LLM이 생성한 마크다운 파일 디렉터리. 요약·엔티티 페이지·개념 페이지·비교·개요·종합. 이 계층은 **LLM이 전적으로 소유**한다. 페이지를 만들고, 새 소스가 오면 갱신하고, 상호참조를 유지하며 일관성을 지킨다. 사람은 읽고 LLM이 쓴다.
3. **스키마 (Schema)** — 위키가 어떻게 구조화되는지, 규칙은 무엇인지, Ingest·Query·유지보수 시 어떤 워크플로를 따를지 LLM에게 알려주는 문서 (Claude Code라면 `CLAUDE.md`, Codex라면 `AGENTS.md`). **핵심 설정 파일**이다. 이것이 LLM을 범용 챗봇이 아닌 **규율 있는 위키 관리자**로 만든다. 도메인에 맞게 사람과 LLM이 함께 진화시킨다.

## 5. 세 가지 운영: Ingest · Query · Lint

### Ingest (수집)

새 소스를 raw 모음에 넣고 LLM에게 처리를 지시합니다. 예시 흐름.

1. LLM이 소스를 읽는다.
2. 핵심 요점을 당신과 논의한다.
3. 위키에 요약 페이지를 쓴다.
4. 인덱스를 갱신한다.
5. 관련 엔티티·개념 페이지를 위키 전반에 걸쳐 갱신한다.
6. 로그에 항목을 덧붙인다.

> 📌 소스 하나가 **10~15개 위키 페이지**를 건드릴 수 있다. 한 번에 하나씩 넣으며 관여할 수도(요약 확인·강조점 지시), 감독을 줄여 여러 소스를 배치로 넣을 수도 있다. 자기 스타일에 맞는 워크플로를 만들고 **스키마에 문서화**해 다음 세션에서 재사용한다.

### Query (질의)

위키를 대상으로 질문합니다. LLM은 관련 페이지를 검색·읽고 **인용과 함께** 답을 종합합니다. 답 형식은 질문에 따라 마크다운 페이지, 비교 표, 슬라이드(Marp), 차트(matplotlib), 캔버스 등 다양합니다.

> 💡 중요한 통찰: **좋은 답은 다시 위키에 새 페이지로 파일링할 수 있다.** 요청한 비교, 분석, 발견한 연결 고리는 가치 있는 자산이므로 채팅 기록 속으로 사라지게 두면 안 된다. 이렇게 하면 당신의 탐구도 수집한 소스처럼 지식베이스에 **복리로 쌓인다**.

### Lint (점검)

주기적으로 위키의 건강 상태를 점검하게 합니다. LLM이 찾는 것.

- 페이지 간 **모순**
- 최신 소스가 뒤집은 **낡은 주장**
- 들어오는 링크가 없는 **고아 페이지(orphan)**
- 언급되지만 자기 페이지가 없는 **중요 개념**
- 빠진 **상호참조**
- 웹 검색으로 채울 수 있는 **데이터 공백**

LLM은 조사할 새 질문과 찾아볼 새 소스를 제안하는 데 능합니다. 이 과정이 위키가 커져도 건강을 유지시킵니다.

## 6. 위키를 항해하는 두 파일: index.md · log.md

위키가 커질 때 LLM(과 사람)이 길을 찾도록 돕는 두 특수 파일이 있습니다. 목적이 다릅니다.

| 파일 | 성격 | 역할 |
|------|------|------|
| **index.md** | 내용 지향 | 위키 전체 카탈로그. 각 페이지를 링크·한 줄 요약·(선택)메타데이터와 함께 카테고리별로 정리. 매 Ingest마다 갱신. Query 시 LLM이 **먼저 인덱스를 읽어** 관련 페이지를 찾고 파고든다. |
| **log.md** | 시간순 | append-only 기록. 무엇을 언제 했는지(Ingest·Query·Lint) 남긴다. 타임라인을 주고 최근에 무슨 일이 있었는지 LLM이 파악하게 한다. |

> 🛠️ 팁: log의 각 항목을 일관된 접두어로 시작하면(`## [2026-04-02] ingest | 기사 제목`) 유닉스 도구로 파싱할 수 있다. `grep "^## \[" log.md | tail -5` → 최근 5개 항목.

인덱스 우선 방식은 놀랍게도 **중간 규모(~100 소스, 수백 페이지)** 까지 잘 동작하며, 임베딩 기반 RAG 인프라 없이도 충분합니다.

## 7. 선택 사항: CLI 도구와 검색

규모가 커지면 LLM이 위키를 더 효율적으로 다루도록 작은 도구를 만들 수 있습니다. 가장 명백한 것은 **위키 페이지 검색 엔진**입니다.

- 소규모에서는 인덱스 파일로 충분하다.
- 커지면 제대로 된 검색이 필요하다. [qmd](https://github.com/tobi/qmd) — 마크다운용 로컬 검색 엔진. BM25/벡터 하이브리드 + LLM 재랭킹, 전부 온디바이스. CLI(LLM이 셸로 호출)와 MCP 서버(네이티브 도구로 사용) 모두 제공.
- 더 단순하게, LLM과 함께 나이브한 검색 스크립트를 바이브 코딩해도 된다.

## 8. Obsidian 실전 팁

<details>
<summary>Obsidian을 위키 뷰어/IDE로 쓸 때의 실용 팁 (펼치기)</summary>

- **Web Clipper**: 웹 기사를 마크다운으로 변환하는 브라우저 확장. 소스를 raw 모음에 빠르게 넣기 좋다.
- **이미지 로컬 다운로드**: Settings → Files and links에서 "Attachment folder path"를 고정 디렉터리(예: `raw/assets/`)로. Hotkeys에서 "Download attachments for current file"를 단축키(예: Ctrl+Shift+D)에 바인딩. 클리핑 후 단축키를 누르면 이미지가 로컬로 저장돼 LLM이 직접 참조 가능. (단, LLM은 인라인 이미지가 든 마크다운을 한 번에 못 읽으므로, 텍스트를 먼저 읽고 이미지를 따로 보게 하는 우회가 필요하다.)
- **그래프 뷰**: 위키의 형태를 보는 최고의 방법. 무엇이 무엇과 연결됐는지, 어떤 페이지가 허브이고 어떤 게 고아인지 한눈에.
- **Marp**: 마크다운 기반 슬라이드 포맷. 위키 내용에서 바로 발표 자료 생성.
- **Dataview**: 페이지 frontmatter(YAML: 태그·날짜·소스 수 등)에 질의해 동적 표·목록 생성.
- **git**: 위키는 그냥 마크다운 git 저장소다. 버전 이력·브랜치·협업을 공짜로 얻는다.

## 9. 왜 이 방식이 통하는가 (Memex 연결)

지식베이스 유지의 지겨운 부분은 읽기나 사고가 아니라 **기록(bookkeeping)** 입니다 — 상호참조 갱신, 요약 최신화, 모순 표시, 수십 페이지의 일관성 유지.

> 🧠 사람이 위키를 포기하는 이유는 **유지보수 부담이 가치보다 빠르게 커지기** 때문이다. LLM은 지루해하지 않고, 상호참조 갱신을 잊지 않으며, 한 번에 15개 파일을 건드릴 수 있다. **유지 비용이 0에 가까워지므로** 위키가 계속 유지된다.

이 아이디어는 정신적으로 **Vannevar Bush의 Memex(1945)** 와 닿아 있습니다. 문서 사이의 연상적 경로(associative trails)를 갖춘, 개인적이고 능동적으로 큐레이션된 지식 저장소. Bush의 비전은 오늘날의 웹보다 이 패턴에 더 가까웠습니다 — 사적이고, 능동적으로 큐레이션되며, 문서 사이의 **연결이 문서만큼 가치 있는** 저장소. 그가 풀지 못한 부분은 "누가 유지보수를 하는가"였고, 이제 그 몫을 LLM이 맡습니다.

## 10. 시작하는 법 — 추상적 패턴, 구체는 당신 몫

이 노트는 **의도적으로 추상적**입니다. 구현이 아니라 아이디어를 설명합니다. 정확한 디렉터리 구조, 스키마 규칙, 페이지 포맷, 도구는 모두 당신의 도메인·취향·선택한 LLM에 달려 있습니다.

- 위에 언급된 모든 것은 **선택적·모듈식**이다. 쓸모 있는 것만 고르고 나머지는 무시하라.
- 소스가 텍스트뿐이면 이미지 처리는 아예 필요 없다.
- 위키가 작으면 인덱스 파일만으로 충분하고 검색 엔진은 필요 없다.
- 슬라이드가 필요 없으면 마크다운 페이지만 써도 된다.

> 🚀 올바른 사용법: 이 문서를 당신의 LLM 에이전트에게 공유하고, **함께** 당신에게 맞는 버전을 인스턴스화하라. 이 문서의 유일한 임무는 패턴을 전달하는 것이며, 나머지는 LLM이 알아서 채운다.
