# 갱신 비용 0의 자가 운영 포트폴리오 — 마크다운 vault를 단일 SSOT로

> 🗂️ 이력서·LinkedIn·개인 사이트·FAQ·AI 챗봇처럼 흩어진 9개 프로필을 매번 손으로 맞추는 대신, **마크다운 knowledge base(vault) 한 곳을 단일 진실 원본(SSOT)** 으로 두고 나머지를 전부 자동 파생시킨 인프라 정리 노트입니다. vault에 마크다운 한 장을 push하면 → LLM ETL이 정규화된 SiteData로 변환하고 → 사이트 카피·페르소나 뷰·FAQ·AI 챗봇이 한꺼번에 갱신됩니다. 신뢰성은 3단 폴백으로, 안정성은 권한 4분할로 지탱합니다. 임한솔의 "AI 포트폴리오 만들기 (1)"을 재구성했습니다.

## 1. 진짜 비싼 비용은 "어디를 고쳐야 하지?"

면접 전날 새벽에 이력서 한 줄을 고치면, LinkedIn이 어긋나고, 그걸 고치면 개인 사이트가 또 어긋난다. 구직자에게 진짜 비싼 비용은 한 줄을 수정하는 시간이 아니라 **"매번 어디를 고쳐야 하는지 머릿속에서 다시 떠올리는 인지 부하"** 다.

원문 저자가 세어본 프로필은 9개다.

- 이력서 PDF · LinkedIn · GitHub 프로필 · 회사 명함
- 사이트 포트폴리오 뷰 · 사이트 FAQ · AI 챗봇 답변 컨텍스트
- 자기소개서 템플릿 · 면접 자기소개 대본

새 회사·새 프로젝트·새 글이 생길 때마다 이 9곳을 머리로 돌려보고 어디를 손댈지 결정해야 한다. 부하가 크니 결국 어딘가는 누락된다. 실제로 인터뷰에서 "사이트의 그 프로젝트를 더 설명해 달라"는 질문을 받았는데 그 프로젝트가 사이트에 없었던 적이 있었다. 갱신 비용이 크면 새 글을 쓰기조차 싫어진다.

## 2. 해법 — source of truth를 한 곳으로

핵심 결심은 단순하다. **9개 중 한 곳을 "원본"으로 정하고, 나머지 8곳은 거기서 자동으로 따라오게 만든다.**

원본 자리에는 마크다운으로 된 personal knowledge base(vault)를 앉힌다. Obsidian 스타일의 객체 지향 폴더 구조로, 객체 타입마다 폴더를 분리한다.

| 폴더(객체 타입) | 예시 | 현재 누적 |
|---|---|---|
| `organizations` | 다녀온 회사 | 14곳 |
| `projects` | 완료한 프로젝트 | 13개 |
| `artifacts` | 글·발표·산출물 | 11종 |
| `people` / `concepts` / `places` / `events` / `action log` | 인물·개념·장소·이벤트·활동 로그 | — |

그 위에 블로그 1,008편, Medium 49편, LinkedIn 174건, 단행본 1권이 별도 레이어로 봉인되어 있다.

> ✍️ 갱신 규칙이 이렇게 바뀐다: "새 회사가 생기면 `organizations`에 **마크다운 한 장**을 더한다. 그게 끝이다. 나머지 8곳 프로필은 거기서 따라온다."

## 3. 마크다운 한 장이 사이트 전체를 바꾸는 파이프라인

vault에 마크다운을 commit·push하면 메인 repo의 빌드 워크플로가 깨어나 7단계를 돈다.

1. vault의 변경을 읽는다.
2. **LLM ETL** — vault를 정규화된 사이트 데이터(SiteData)로 변환한다.
3. 변환된 SiteData를 vault submodule에 commit·push로 다시 박아 넣는다.
4. 빌드 타임 폴백 스냅샷을 만든다.
5. **zod 스키마**로 검증한다.
6. lint를 돈다.
7. 빌드해서 호스팅 플랫폼에 배포한다.

가장 흥미로운 건 2단계다. LLM이 사람이 쓴 마크다운(객체 인스턴스들)을 읽고 구조화된 뷰 하나로 정렬해 낸다. 정렬 대상은 다음 영역들이며, 이 묶음이 **`ObjectView` 한 인스턴스**로 응축된다.

```text
identity · pillars · personas · viewHeaders
portfolioCopy (home / hire / collab / builder / curious / ask)
career · education · certifications · languages · publications · faq
```

## 4. 한 인스턴스에서 모든 결과물이 파생된다

이 `ObjectView` 한 인스턴스가 **사이트 홈페이지 전체의 단일 SSOT** 다. 아래 결과물이 전부 같은 인스턴스의 서로 다른 시각화일 뿐이다.

- **페르소나 뷰 4종** — `hire`(채용) / `collab`(협업) / `builder`(개발) / `curious`(탐색)
- 이력·자격·언어·publications 정보 묶음
- **FAQ**
- **AI 챗봇** — 같은 인스턴스를 컨텍스트로 참조

> 🤖 챗봇을 따로 학습시키는 단계가 없다. 운영자가 챗봇을 가르치는 게 아니라, **챗봇이 운영자의 글을 함께 보는 구조**다. 새 회사를 추가하는 즉시 챗봇은 그 회사에 대해 사실 기반으로 답하기 시작한다.

정형 질문은 **FAQ fast-path**로 즉답하고, 자유 질문만 **RAG 파이프라인**으로 흘린다. 이 한 가지 패턴이 갱신 비용을 0에 수렴시키는 핵심이다. **사람은 마크다운을 쓰고, 사이트와 챗봇은 그 마크다운을 본다.** 인과가 한 방향으로만 흐른다.

## 5. 신뢰성 — 3단 폴백으로 서비스는 죽지 않는다

"파이프라인이 어디선가 깨지면 포트폴리오도 깨지지 않나?"에 대한 답이 3단 폴백이다. 데이터 로더는 아래를 **순서대로** 시도하고, 첫 성공 결과를 **5분간 캐시**한다.

| 단 | 소스 | 성격 |
|---|---|---|
| 1단 | 원격 **Blob**의 최신 SiteData | 항상 최신 |
| 2단 | 로컬 **vault submodule**의 SiteData | 빌드에 포함된 사본 |
| 3단 | 빌드 시점에 **코드에 박아 둔 폴백 스냅샷** | 최후의 보루 |

원격 Blob 접근 토큰도 **환경변수 3종을 순서대로** 폴백한다. 한 토큰이 만료돼도 다른 토큰이 받아낸다.

> 🛡️ 효과는 분명하다: vault repo의 CI가 죽어도, 메인 repo의 CI가 죽어도, Blob이 잠시 불통이어도 사이트는 살아 있다. 정확히는 **"마지막으로 살아 있던 시점의 데이터"** 로 계속 답한다. 어느 단이 흔들려도 5분 뒤의 사용자는 그 흔들림을 모른다.

## 6. 안정성 — 권한 4분할 (사람 · AI 세션 · CI)

vault에 글을 쓰는 주체는 한 명이 아니다. 세 주체가 같은 vault submodule에 commit·push로 수렴한다.

- 사람의 수동 편집 (Obsidian으로 직접)
- **Claude / Cowork 세션**을 통한 편집
- 메인 repo의 빌드 워크플로 (LLM ETL이 SiteData 한 파일만 자동 갱신)

단, Blob 업로드는 이 셋 중 누구도 직접 하지 않는다. Blob 동기화는 vault repo(`hsol-info-blob`)의 자체 GitHub Actions가 push 이벤트로 트리거되어 담당한다. 정리하면 권한이 4분할된다.

| 권한 | 주체 |
|---|---|
| (a) vault에 글쓰기 | **사람 + Claude 세션** |
| (b) Blob 업로드 | vault repo 자체 CI |
| (c) SiteData 정규화 · submodule 갱신 | 메인 repo CI |
| (d) 빌드 · 배포 | 메인 repo CI + 호스팅 플랫폼 |

> 🔐 권한이 흩어져 있다는 사실 자체가 안정성을 만든다. 한 권한이 흔들려도 다른 권한이 사이트를 살린다. 사용자가 손대는 곳은 (a)뿐 — **마크다운 한 장 + git push**가 전부다. "1인 SaaS의 운영팀은 사람 한 명과 AI 한 세션"이라는 말이 비유가 아니라 시스템에 그대로 박혀 있다.

## 7. ASIS → TOBE 비교

| 이벤트 | ASIS (수정 = 서너 곳 반영) | TOBE (수정 = 마크다운 한 장) |
|---|---|---|
| 새 회사 입사·퇴사 | 이력서·LinkedIn·포트폴리오 뷰·챗봇 컨텍스트 4곳 | `organizations`에 md 1장 → push |
| 새 프로젝트 완료 | 이력서·사이트 카피·면접 대본 3곳 | `projects`에 md 1장 → push |
| 새 글·발표·산출물 | 사이트 publications·LinkedIn·자소서 템플릿 3곳 | `artifacts`에 md 1장 → push |
| FAQ 답변 변경 | 자기소개 대본·면접 답변·사이트 FAQ·챗봇 4곳 | ObjectView `faq` 한 항목 수정 → push |
| 프로필 톤 조정 | 사이트 카피 전체·자기소개·면접 대본 | 페르소나 운영 매뉴얼 ObjectView 1곳 → push |
| 면접 직전 점검 | 이력서·사이트·LinkedIn 어긋난 곳 찾기 | 그냥 잔다 (사이트가 이미 최신) |

표에 안 보이는 가장 큰 차이는 인지 부하다. ASIS에선 매번 "어디를 고쳐야 하지?"가 먼저 떠올랐고, TOBE에선 그 질문 자체가 사라졌다.

## 8. 따라 하기 위한 최소 재현 단위

거창해 보여도 **한 명이 운영 가능한 최소 단위**로 압축된다: vault 1개 · 메타-온톨로지 6종 · LLM ETL 1단 · 3단 폴백 · 권한 4분할.

<details>
<summary>5개 부품 상세 펼치기</summary>

1. **vault** — 마크다운 knowledge base 한 곳. Obsidian이든 단순 git repo든 무관. 객체 타입별(회사·프로젝트·산출물·개념)로 폴더를 나누면 출발선이 생긴다.
2. **메타-온톨로지 선언** — 객체 타입·링크 타입·액션 타입을 명시적으로 적어 둔다. 스키마가 머릿속에만 있으면 6개월 뒤 무너진다. vault 안에 별도 폴더로 데이터 모델을 두면 미래의 나와 미래의 AI 세션이 같은 규칙으로 쓴다.
3. **LLM ETL** — vault를 정규화된 SiteData로 변환하는 한 단계. Claude·GPT 어느 쪽이든 작동한다. 단, 응답이 토큰 예산 초과로 잘리면 **억지로 파싱하지 말고 실패 덤프로 보존**하라. 잘린 출력을 파싱하면 사이트가 잘못된 데이터로 배포된다.
4. **3단 폴백** — 원격 캐시 · 로컬 vault · 코드 내 폴백을 직렬로 시도. 5분 캐시 TTL, 토큰도 3종 폴백으로 환경 차이를 흡수.
5. **권한 분리** — 글쓰기·배포·외부 동기화 권한을 서로 다른 주체에 나눈다. 한 워커가 모든 권한을 쥐면 하나의 실수가 사이트를 날린다.

## 9. 무엇이 사라졌는가

- 벼락치기 이력서 다듬기 → 사이트가 항상 최신, 이력서는 그 한 시점 스냅샷일 뿐.
- 별도 "인터뷰용 자기소개" 페이지 작성 → 페르소나 뷰 `hire`가 대신한다.
- 미팅 전 자기소개 → 채용 담당자가 ChatDock에서 먼저 질문하고 챗봇이 vault의 ObjectView를 참조해 답한다.
- 누락 사고 → 그 뒤로 다시 일어나지 않았다.

가장 큰 변화는 시간 절약이 아니라 **"어디를 고쳐야 하지?"라는 질문이 매일 머릿속에서 사라진 것**이었다. 구직자에게 가장 좋은 자기 표현은 이력서가 아니라 **누적된 글**이고, vault가 그 글을 받는 그릇이며 사이트는 한 단면일 뿐이다. 포트폴리오는 갱신 비용이 0에 수렴해야 살아남는다.
